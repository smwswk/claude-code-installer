# 直达根源

## 描述

第一性原理决策切片器。识别输入中"中间件/多余动作"的元层本质，用第一性原理拆解为"真实功能 → 真实目标 → 现代短路径 → 塔勒布否决 → 起草执行物"，用户确认后执行 1%。输入一句话困惑，输出 99% 完成度的决策文档。

**核心设计**: 用户完成 99%，用户只做最后 1% 的确认。

## 触发条件

- 用户表达选择困难、困惑、"该不该..."、"还要不要..."
- 用户说"直达根源"、"决策切片"、"第一性原理"
- 用户描述一个行动/决策，隐含"这是惯性还是必要"的疑问

## 工作流程

### Phase 0: 定义复核（规则 9）

跑诊断前，必须先问用户（或自我推断后向用户确认）：

> "你这件事在你这里的真实功能是 X 还是 Y？"

不要直接套用行业默认定义。同一事物在不同人手里功能完全不同（例：播客 = toC 内容产品 vs toB BD 渠道）。

**如果用户重新定义了事物的功能，触发元规则：重定义触发重跑。**

### Phase 1: 第一性原理拆解

识别输入中"中间件"的真实功能。追问三层：

1. 这件事名义上解决什么？
2. 它实际上解决什么？（用户画像 / 场景 / 时代背景下）
3. 这些子问题在今天还存在吗？

输出一个"功能拆解表"。

### Phase 2: 目标对齐

从用户 memory 和对话上下文推断真实目标。不要默认"流量""影响力"这种泛泛目标。

关键问题：用户当前的真实状态是什么？（如"摆烂""高能量""过渡期"）状态会改变决策。

如果用户的提问隐含"逃避合理化"倾向（低能量状态下提出"砍掉/停掉"），先标记为"待验证的逃避"，后续 Phase 4 规则 8 处理。

### Phase 3: 替代路径

找现代等价短路径。核心判断：10 倍杠杆闸。

- 有没有 AI / 自动化 / 新分发路径可以压缩中间件？
- 投入产出比是否达到 10 倍？
- 是否有"一份母版 → 多平台分发"的可能性？

### Phase 4: 否决器（塔勒布七原则 + 2 条元规则）

任何方案强制过一遍。有否决就明说，给出具体理由。用户可反驳。

1. **反脆弱** — 有上行无下行？失败成本可控？
2. **杠铃** — 极保守 + 极冒险的两端配置？不是中庸平均？
3. **否定法** — 减少负面 vs 追求正面？
4. **少数派** — 是否利用你的真实优势（不被算法/共识覆盖的角落）？
5. **延迟归因** — 真实回报周期是否与你的耐心匹配？
6. **工具冗余** — 是否依赖脆弱的单点？
7. **10 倍影响力闸** — 这件事能撬动 10 倍的什么？撬不动就否决。
8. **逃避合理化检测** — 当用户在低能量状态提出"砍/停/退出"时，先反问"是减负还是逃避？"再决定路径。
9. **定义复核** — 事物的功能定义是否被用户本人明确？是否经得起重定义？

**元规则: 重定义触发重跑**
如果用户在对话中重新定义了某事物的核心功能（如"播客从 toC 内容变成 toB BD 渠道"），必须**从头完整重跑**整个 7 步流程，不得套用之前的中间结论。

### Phase 5: 起草执行物

根据诊断结论，起草具体的行动物。常见类型：

- 决策记录文档（`~/Documents/decisions/{日期}-{主题}.md`）
- 邮件/话术草稿
- 退订/退会清单
- 日历清理项
- 执行 checklist

**交付标准**: 用户拿到手可以直接复制使用，不需要再思考"怎么写"。

### Phase 6: 用户确认

唯一需要用户做的事。用 Yes/No 或简短确认。

### Phase 7: 执行 1%

用户确认后，执行文件保存、邮件发送、日历删除等实际操作。第一版只负责写到 `~/Documents/decisions/`，由用户手动复制执行。

## 输出格式

每次运行用统一的表格输出：

```
| 步骤 | 输出 |
|---|---|
| ①真实功能 | ... |
| ②目标 | ... |
| ③短路径 | ... |
| ④否决器 | 逐项过一遍，有否决标 ✗，通过标 ✓ |
| ⑤建议 | 具体可执行的结论 |
| ⑥起草 | 文件路径 + 内容摘要 |
| ⑦ | ← 用户确认 |
```

## 状态识别

工具必须能识别用户当前状态并调整输出：

- **高能量期** → 激进扩张建议
- **摆烂/低能量期** → 轻量级建议，不追量，协议式执行
- **过渡期** → 维持型建议，不做大决策

状态从 memory（如"案件待办""小红书发布进度""摆烂"等关键词）和对话上下文推断。
